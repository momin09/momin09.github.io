## 배경

로컬에서 LLM을 돌리다 보면 "왜 컨텍스트를 32K에서 128K로 올렸더니 갑자기 느려졌지?" 같은 순간이 온다.

이때 가장 먼저 이해해야 하는 개념이 **KV 캐시(Key-Value Cache)** 다.  
컨텍스트 길이와 KV 캐시는 따로 노는 설정이 아니라, 사실상 같은 현상을 논리적 관점과 물리적 관점에서 부르는 이름에 가깝다.

## 핵심 개념

### 1. KV 캐시란 무엇인가

LLM이 토큰을 하나씩 생성할 때, 이미 처리한 토큰들의 중간 계산 결과를 저장해 두는 메모리가 **KV 캐시**다.

Transformer의 attention은 새 토큰을 생성할 때마다 이전 토큰들과의 관계를 계산해야 한다.

- 토큰 1 생성: 1개 토큰과 attention
- 토큰 2 생성: 2개 토큰과 attention
- 토큰 N 생성: N개 토큰과 attention

이걸 매번 처음부터 다시 계산하면 비용이 급격히 커진다.

그래서 각 토큰을 한 번 처리할 때 생성되는 **Key(K)** 와 **Value(V)** 를 저장해 두고 재사용한다.  
새 토큰이 들어오면 자기 자신의 Query만 계산하고, 이전 토큰의 K/V는 캐시에서 꺼내 쓴다.

이 저장 공간이 바로 KV 캐시다.

### 2. 컨텍스트 길이와 KV 캐시의 관계

"컨텍스트 32K"의 의미는 단순히 긴 문서를 읽을 수 있다는 뜻만이 아니다.

> 최대 32,768개 토큰의 K와 V를 저장할 수 있는 공간을 준비한다는 뜻

즉:

- **컨텍스트 길이**는 논리적 한도
- **KV 캐시 크기**는 그 한도를 실제 메모리로 구현한 결과

Ollama, llama.cpp 같은 런타임은 보통 KV 캐시를 **선할당(pre-allocate)** 한다.  
실제 대화가 짧아도, 설정한 `num_ctx` 기준으로 메모리를 먼저 잡는 경우가 많다.

### 3. KV 캐시 크기 계산

대략적인 공식은 다음과 같다.

```text
KV 캐시 크기 = 2 × num_layers × num_kv_heads × head_dim × context_length × bytes_per_element
              │
              └─ K와 V 두 개
```

Gemma 4 26B를 예시로 잡으면 대략 이런 식이다.

- layers: ~60
- kv_heads: ~8
- head_dim: 256
- context: 32,768
- fp16: 2 bytes

```text
2 × 60 × 8 × 256 × 32768 × 2 ≈ 16 GB
```

이건 단순 이론치고, 실제 런타임에서는 하이브리드 attention, sliding window, KV 양자화 같은 최적화가 들어가서 더 작아진다.  
그래도 중요한 건 **컨텍스트 길이에 따라 KV 캐시가 거의 선형적으로 커진다**는 점이다.

### 4. 왜 모델 간에는 공유되지 않는가

KV 캐시에 저장되는 값은 각 모델의 가중치를 통과한 결과다.

```text
입력 토큰 → [모델 A의 W_k, W_v] → K_A, V_A
입력 토큰 → [모델 B의 W_k, W_v] → K_B, V_B
```

같은 문장을 넣어도 모델이 다르면:

- 레이어 수가 다르고
- KV head 수가 다르고
- 차원 수가 다르고
- 임베딩 공간과 수치값 자체가 다르다

그래서 KV 캐시는 모델별로 완전히 독립적이다.

| 상황 | KV 캐시 공유 여부 |
|---|---|
| 같은 모델, 같은 요청 | 재사용 가능 |
| 같은 모델, prefix가 같은 요청 | 런타임에 따라 부분 공유 가능 |
| 다른 모델 | 공유 불가 |

## 실전에서 중요한 점

### 1. 컨텍스트를 늘리면 VRAM도 같이 늘어난다

| 컨텍스트 | KV 캐시 (비례) | 의미 |
|---|---|---|
| 8K | 1x | 가장 보수적 |
| 32K | ~4x | 일반적인 기본값 |
| 128K | ~16x | 긴 문서 처리 가능, VRAM 부담 큼 |
| 256K | ~32x | 로컬 환경에서는 매우 무거움 |

핵심은 단순하다.

> 컨텍스트를 늘린다 = KV 캐시 공간을 더 예약한다 = VRAM을 더 먹는다

### 2. 가장 위험한 구간은 "조용히 느려지는" 순간이다

VRAM이 한계를 넘으면 Windows 환경에서는 시스템 RAM을 가상 VRAM처럼 쓰는 fallback이 걸릴 수 있다.

이때 문제는:

- 명확한 OOM이 안 나고
- 그냥 점점 느려지고
- tok/s가 눈에 띄게 무너진다는 점이다

특히 모델 로드 직후는 멀쩡한데, 대화가 길어지면서 KV 캐시가 커져서 중간부터 갑자기 느려지는 경우가 있다.

### 3. 대응 전략

#### 즉시 조치

1. `num_ctx`를 낮춘다.
2. 브라우저, Discord, Electron 앱처럼 VRAM을 먹는 백그라운드 앱을 정리한다.
3. 가능하면 KV 캐시 양자화를 사용한다.

예:

```bash
OLLAMA_FLASH_ATTENTION=1
OLLAMA_KV_CACHE_TYPE=q8_0
```

#### 진단

4. 스필을 허용하지 않는 방향으로 설정해, 조용한 성능 저하 대신 명확한 한계를 확인한다.

#### 설계

5. 항상 최대 컨텍스트를 잡지 말고, 실제 용도에 맞춰 `num_ctx`를 나눈다.
6. 큰 모델은 필요할 때만 로드하고, 일상 작업은 더 작은 모델로 분리한다.

## 요약

- KV 캐시는 LLM의 이전 토큰 계산 결과를 저장하는 메모리다.
- 컨텍스트 길이는 KV 캐시의 최대 크기를 결정하는 설정이다.
- 컨텍스트를 키우면 KV 캐시도 같이 커지고, VRAM 사용량도 늘어난다.
- VRAM 한계를 넘기면 OOM보다 먼저 "조용한 성능 저하"가 올 수 있다.
- 로컬 추론에서는 모델 크기만큼이나 **컨텍스트 설정**이 성능과 안정성에 큰 영향을 준다.
