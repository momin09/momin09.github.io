## 환경

이 글은 아래와 같은 전제를 기준으로 본다.

- Kubernetes 기반 워크로드
- EKS 환경
- Datadog Agent 또는 Datadog Operator/Helm 기반 배포
- 애플리케이션에서 Datadog tracer 라이브러리 사용

실제 환경마다 배포 방식은 다를 수 있지만, 트러블슈팅 순서는 거의 비슷하다.

## 문제 상황

APM을 붙일 때 자주 만나는 증상은 대체로 아래 셋 중 하나다.

1. **Trace가 아예 안 들어온다**
2. **서비스는 보이는데 span이 비정상적으로 적다**
3. **데이터는 들어오지만 service/env/version 태그가 정리되지 않아 화면이 지저분하다**

이 셋을 한꺼번에 보려고 하면 오히려 헷갈린다.  
그래서 먼저 문제를 아래처럼 나누는 게 좋다.

- Agent가 trace를 받을 준비가 되었는가
- 애플리케이션이 trace를 보내고 있는가
- Datadog에서 올바른 서비스로 묶이고 있는가

## 설정 체크포인트

### 1. Agent에서 APM이 활성화되어 있는가

가장 먼저 볼 건 Datadog Agent 쪽 설정이다.

- APM 기능이 활성화되어 있는지
- trace 수집 포트가 열려 있는지
- DaemonSet/Pod가 정상 상태인지

Helm이나 Operator를 쓰는 경우에도 핵심은 같다.  
`APM이 켜져 있지 않으면` 애플리케이션이 아무리 잘 보내도 Datadog 쪽에서 받지 못한다.

### 2. 애플리케이션 tracer가 실제로 초기화되었는가

다음은 애플리케이션 쪽이다.

- tracer 라이브러리가 애플리케이션 시작 시점에 로드되는지
- 자동 instrumentation이 실제로 적용되었는지
- 프레임워크별 초기화 순서가 맞는지

이 구간은 "설정은 넣었는데 trace가 하나도 안 찍히는" 문제의 핵심 원인인 경우가 많다.

### 3. Agent와 애플리케이션 사이 네트워크가 열려 있는가

특히 Kubernetes에서는 네트워크 경로를 빼먹기 쉽다.

- 애플리케이션이 Agent 주소를 올바르게 바라보는지
- `8126` 포트로 전송 가능한지
- NetworkPolicy나 보안 설정으로 막히지 않았는지

설정값이 맞아 보여도 실제 통신이 막혀 있으면 trace는 올라오지 않는다.

### 4. service / env / version 태그가 정리되어 있는가

APM이 "보인다"와 "운영에 쓸 수 있다"는 다른 문제다.

최소한 아래 태그는 일관되게 맞추는 편이 좋다.

- `service`
- `env`
- `version`

이 세 값이 흔들리면:

- 같은 서비스가 여러 개로 쪼개져 보이고
- 배포 버전 비교가 어려워지고
- 환경별 분석도 불편해진다

## 트러블슈팅 순서

### 1단계. Agent가 살아 있는지 확인

가장 먼저 Agent Pod 상태와 설정을 확인한다.

- Pod가 Running인지
- APM 관련 옵션이 켜져 있는지
- 로그에 trace 수신 관련 오류가 없는지

여기서 막히면 애플리케이션을 볼 필요가 없다.

### 2단계. 애플리케이션이 trace를 전송하는지 확인

그다음은 애플리케이션 로그나 tracer 디버그 로그를 본다.

- tracer 초기화 로그가 있는지
- 샘플 요청 이후 span 생성 흔적이 있는지
- 환경 변수나 설정 파일이 실제 반영되었는지

### 3단계. 네트워크 경로 확인

설정과 코드가 다 맞아도, 실제로 Agent까지 닿지 않으면 끝이다.

이 단계에서는:

- Agent endpoint 주소
- 포트
- Pod 간 통신 가능 여부

를 확인한다.

### 4단계. Datadog UI에서 서비스 묶임 확인

마지막으로 Datadog UI에서:

- 서비스명이 의도한 대로 들어오는지
- env/version 분리가 맞는지
- span 수가 너무 적지는 않은지

를 본다.

이 단계에서 문제가 있으면 "수집 실패"보다는 "태깅/설정 정합성 문제"인 경우가 많다.

## 운영 팁

### 1. APM 도입 시 최소 태그 규칙부터 정한다

서비스 이름 규칙을 먼저 정해 두면 나중에 훨씬 편하다.

예:

- `service`: 애플리케이션 단위
- `env`: `prod`, `stg`, `dev`
- `version`: 배포 단위 식별값

### 2. 설정과 트러블슈팅을 분리해서 기록한다

설정 문서와 장애 대응 메모가 섞이면 나중에 재사용하기 어렵다.

그래서 문서를 나눌 때도:

- **설정 가이드**
- **트러블슈팅 가이드**

형태로 분리하는 편이 좋다.

### 3. "안 보인다"를 하나의 문제로 취급하지 않는다

APM이 안 보이는 문제는 사실 하나가 아니라 여러 문제의 묶음이다.

- 수집이 안 되는 문제
- 전송이 안 되는 문제
- 분류가 안 되는 문제

이걸 나눠서 보면 훨씬 빨리 해결된다.

## 요약

- Datadog APM 트러블슈팅은 `Agent`, `애플리케이션 tracer`, `네트워크`, `태깅` 네 축으로 나누어 보면 된다.
- 가장 빠른 확인 순서는 `수집 준비 여부 → 전송 여부 → UI 반영 여부`다.
- 문서 포맷도 이 흐름을 따르는 편이 나중에 다시 보기 좋다.
- 앞으로 Observability 글은 `트러블슈팅형`, AI 글은 `개념 정리형`으로 가져가면 블로그 전체 스타일이 더 안정적으로 맞아진다.
